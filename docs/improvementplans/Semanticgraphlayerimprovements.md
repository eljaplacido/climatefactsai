## RAG design

Your RAG should become **claim-first** rather than article-first, because the platform’s core unit of trust is a verified or disputed claim with evidence, not a whole article. The reports also say the current retriever lacks semantic understanding of support, contradiction, and causation, so the next version should retrieve not only “similar” items but also stance-linked and path-linked items. I’d structure retrieval in six stages:

1. Query planner: decompose into entities, geography, time, regulation, company, metric, and intent.
2. Hybrid fetch: retrieve from separate indexes for claims, evidence spans, articles, entities, source profiles, corporate disclosures, regulations, projections, and prior session memory.
3. Graph expansion: hop outward from seed entities/claims to supporting, contradicting, regulating, or time-adjacent nodes.
4. Reranking: score with source-tier axes, numeric-grounding status, cross-LLM agreement, recency, geo match, persona/view mode, and stance relevance.
5. Evidence pack assembly: build a compact answer context from evidence spans, not raw long chunks.
6. Deterministic post-check: bind every generated statement to provenance IDs and run numeric/claim-type validators before display.

For indexing, create multiple representations per object: raw text, normalized claim text, structured facets, and an embedding. I would add parent-child chunking for articles, claim/evidence microchunks for verification, and separate embeddings for entity profiles, source profiles, regulations, and time-series metadata so retrieval can support both analysis and visualization cleanly.

## Graph model

Use a property graph operationally, with claims as first-class nodes rather than storing relationships only as edge metadata. That fits your platform better than pure RDF because you need fast traversal, agentic path expansion, explainable k-hop reasoning, and dashboard-friendly graph analytics; you can still export RDF or JSON-LD views later for interoperability and compliance.

A practical node model would include: `Article`, `Claim`, `EvidenceSpan`, `CanonicalEntity`, `EntityMention`, `Source`, `Country`, `Company`, `Metric`, `ProjectionSeries`, `Regulation`, `Framework`, `Event`, `AnalysisRun`, `PromptFingerprint`, `UserSession`, and `MemoryItem`. Key edges should include `ASSERTED_IN`, `SUPPORTED_BY`, `CONTRADICTED_BY`, `MENTIONS`, `SAME_AS`, `ABOUT_COUNTRY`, `ABOUT_COMPANY`, `REGULATED_BY`, `PROJECTS`, `CITES`, `DERIVED_FROM`, `GENERATED_BY`, and `VIEWED_IN_SESSION`.

Two design choices matter most. First, split `EntityMention` from `CanonicalEntity`, so you can resolve “EU,” “European Union,” and “Brussels” into one canonical object while preserving source text; second, attach validity windows, confidence, provenance ID, and extraction method to every edge so the graph can power both reasoning and audit trails. That is the cleanest way to turn the current JSONB shadow graph into a multihop reasoning layer without weakening your trust posture.

## Memory and indexing

For coherent context across the whole platform, I would use a three-part memory model: short-term working memory in Redis, durable episodic memory in Postgres, and semantic long-term memory in the graph plus vector indexes. That maps well to your existing architecture because you already have durable provenance and a rigorously typed agent layer, but you need a consistent memory contract across search, analyze, deep-search, chat, and visualization surfaces.
