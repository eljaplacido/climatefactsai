# Performance Audit â€” 2026-05-18

**Scope**: climatenews / Climatefacts.ai backend (FastAPI), Postgres 16 (pgvector 0.8.1), Celery, Next.js 14 frontend.

## Aggregate picture

The platform is built on a *synchronous* `db.execute_query` (sync SQLAlchemy engine) called from `async def` route handlers â€” every DB hop blocks the FastAPI event loop. On top of that, **almost every aggregate map/green/analytics endpoint runs a per-country query loop instead of one CTE**, multiplying request latency by 2-200Ã—. The route the task brief flagged (`/api/map/compare`) is the most visible offender (4 queries Ã— N countries), but `GET /api/map/country-stats` is worse: it issues `2 Ã— N_countries + 2` queries on every map-page mount and there is no shared cache because routes use a per-process `_cache: Dict` instead of Redis (which is wired up but only used by 4 routes). Open-Meteo enrichment is the single biggest unlock: every article enrichment job hits the Open-Meteo forecast + archive APIs in series with **no caching at all**, despite a `weather_cache` table existing in the schema with zero readers/writers in code â€” backfilling 1000s of articles re-fetches the same per-country coordinates thousands of times. A second-order finding: migrations 014-025 (HNSW, multilingual FTS, `is_synthetic`, `country_indicators`, sessions) are **not applied locally** (`schema_migrations` ends at 013), so `hybrid_rag_service.py` queries `articles.search_tsv` (line 136) against a column that doesn't exist â€” RAG retrieval errors out silently on this environment and the `is_synthetic` partial index in the schema is moot because the local table doesn't have the column either.

**Single biggest unlock**: collapse the per-country fan-out in `country-stats` / `compare` / `green-transition/country` into one CTE (one DB roundtrip instead of 400) and put a Redis-backed 5-minute cache in front of it. That alone moves the map page from ~3-8 s P95 to <300 ms and frees a worker slot during burst traffic.

## Out-of-scope but worth a note (not in top-10)

- **Migrations 014-025 not applied** to the local DB (`schema_migrations` stops at 013, yet code references `articles.search_tsv`, `articles.is_synthetic`, `country_indicators`, `chat_sessions`, etc.). Ops issue, but it means RAG/FTS/synthetic-filter code paths silently raise on this environment and degrade to fallbacks. Run the migration set against local before any real perf benchmark.
- `_fulltext_search` (`api/chat_routes.py:415-428`) uses an **inline** `to_tsvector('english', COALESCE(title,'')||â€¦)` expression that **does not exactly match** the `idx_articles_fts` index expression (cast `'english'::regconfig` differs from the literal `'english'`). `EXPLAIN` confirms a Seq Scan today. Either align the expression byte-for-byte or query `search_tsv` once 018 lands.
- Celery `worker_concurrency=4`, `worker_prefetch_multiplier=4` (`celery_config.py:50-57`) is conservative on a 23-country staggered schedule but fine for current volume; revisit when daily ingest > 500 articles.
- `cost_tracking` / `auth_audit` tables are unindexed on `user_id` / `created_at`-pair lookups â€” out of scope for this audit but flag for a follow-up index pass.
