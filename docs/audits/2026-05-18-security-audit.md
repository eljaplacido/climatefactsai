# Security Audit — 2026-05-18

**Scope:** Residual / new OWASP Top 10 issues after the 2026-05-16 sprint-1 remediation wave (OAuth state CSRF, `email_verified` enforcement, Next 14.2.30 + DOMPurify, SSRF redirect re-validation, URL job-token, PyJWT 2.10.1, stateful sessions + refresh-token rotation + replay detection, revoke-on-password-change).

## Summary

Overall posture is **markedly improved** from sprint 1; the previously-shipped controls all hold up to a focused re-review (refresh rotation is sound, OAuth `email_verified` is enforced both as bool and string, the URL job-token uses HMAC + constant-time comparison, the new manual-redirect SSRF loop re-validates each hop, no hardcoded secrets in the repo). The biggest remaining concern is the gap between **policy and enforcement on operational endpoints**: `SCHEDULER_SECRET` and `CLILENS_CALIBRATION_ADMIN_SECRET` are **only enforced when set** — if either env var is empty/missing in production, the corresponding admin/scheduler endpoints become completely unauthenticated. That, combined with the absent **Content-Security-Policy**, the still-pending **localStorage → httpOnly cookie** migration, the **missing `/privacy` and `/terms` Next.js routes** (the signup form links to dead URLs), and a residual **SSRF TOCTOU** between `_validate_safe_url`'s DNS resolution and `httpx`'s own resolution, defines the queue for sprint 2. No critical (auth-bypass / RCE / direct data-exfil) findings; the High items are all configuration-cliff or compliance-surface issues.

## Notes on items that held up

- **OAuth state CSRF (sprint 1)**: `api/oauth_routes.py:106-119` validates length + URL-safe charset on `state`. The state is generated client-side via `/api/auth/oauth/state` and the server never persists it — so technically the state is unbound to a session. Current control catches blind-CSRF (attacker must inject a state value), but not full session-binding. Sprint 2 nice-to-have: HMAC the state with `OAUTH_STATE_SECRET` server-side and verify on callback.
- **Refresh-token replay detection**: logic at `api/auth_utils.py:191-213` is correct — the race-condition above is the only remaining gap.
- **URL job-token**: HMAC-SHA256, constant-time compare (`api/url_analysis_routes.py:438-450`). Good.
- **DOMPurify**: only one `dangerouslySetInnerHTML` site (`src/frontend/src/components/ArticleDetailTabs.tsx:74`), strict tag/attr allowlist + URI regex. Chat answers go through `ReactMarkdown` (no `rehype-raw`), so untrusted LLM HTML cannot break out.
- **PyJWT 2.10.1 + no python-jose**: `api/auth_utils.py:21,287-295` uses `jwt.InvalidTokenError` (PyJWT base class), no `JWTError` (python-jose). Clean.
- **Admin endpoint auth**: `/api/admin/pipeline/*` (`api/admin_pipeline_routes.py:38-52`) checks `subscription_tier == 'enterprise' or email in ADMIN_EMAILS`. Solid — the gap is only on the secret-header endpoints called out above.
